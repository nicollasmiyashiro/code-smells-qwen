Classes Alternadas

Alternate Classes with Different Interfaces

📖 Definição

Este code smell ocorre quando duas classes fazem essencialmente a mesma coisa, mas têm interfaces diferentes. Você precisa lembrar qual classe usar em cada situação, e o código se torna inconsistente.

Este problema frequentemente surge quando diferentes desenvolvedores criam funcionalidades similares independentemente, ou quando uma classe é estendida com uma interface inconsistente.

⚠️ Problemas Causados

  • Confusão de uso: Desenvolvedores não sabem qual classe usar.
  • Inconsistência: Código usa classes diferentes para mesma finalidade.
  • Duplicação de lógica: Mesma lógica implementada duas vezes.
  • Manutenção dobrada: Mudanças precisam ser feitas em duas classes.
  • Poluição do código: Mais classes do que o necessário.

🔍 Como Identificar

  • Duas classes com nomes similares (Cliente vs ClienteData).
  • Mesmos dados/conceitos com nomes de métodos diferentes.
  • Conversores entre as duas classes (Cliente para ClienteDTO e vice-versa).
  • Comentários como "use esta classe ao invés daquela".
  • Classes criadas em épocas diferentes para propósitos similares.

❌ Código RUIM

❌ Java - Classes com Interfaces Diferentes 
// Classe antiga - criada primeiro
public class Cliente {
    private Long id;
    private String nome;
    private String email;
    
    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }
    
    public String getNome() { return nome; }
    public void setNome(String nome) { this.nome = nome; }
    
    public String getEmail() { return email; }
    public void setEmail(String email) { this.email = email; }
    
    public String getNomeCompleto() {
        return nome.toUpperCase();
    }
}

// Classe nova - criada depois, interface diferente
public class ClienteDTO {
    private Long identifier;
    private String fullName;
    private String emailAddress;
    private boolean active;
    
    public Long getIdentifier() { return identifier; }
    public void setIdentifier(Long identifier) { this.identifier = identifier; }
    
    public String getFullName() { return fullName; }
    public void setFullName(String fullName) { this.fullName = fullName; }
    
    public String getEmailAddress() { return emailAddress; }
    public void setEmailAddress(String emailAddress) { this.emailAddress = emailAddress; }
    
    public boolean isActive() { return active; }
    public void setActive(boolean active) { this.active = active; }
}

// Conversor necessário porque as classes são diferentes
public class ClienteMapper {
    public static Cliente toEntity(ClienteDTO dto) {
        Cliente cliente = new Cliente();
        cliente.setId(dto.getIdentifier());
        cliente.setNome(dto.getFullName());
        cliente.setEmail(dto.getEmailAddress());
        return cliente;
    }
    
    public static ClienteDTO toDto(Cliente cliente) {
        ClienteDTO dto = new ClienteDTO();
        dto.setIdentifier(cliente.getId());
        dto.setFullName(cliente.getNome());
        dto.setEmailAddress(cliente.getEmail());
        dto.setActive(true);
        return dto;
    }
}

✅ Código BOM

✅ Java - Interface Unificada 
// Interface comum
public interface ICliente {
    Long getId();
    String getNome();
    String getEmail();
    boolean isAtivo();
    
    default String getNomeFormatado() {
        return getNome().trim();
    }
}

// Implementação para domínio
public class Cliente implements ICliente {
    private final Long id;
    private final String nome;
    private final String email;
    private boolean ativo;
    
    public Cliente(Long id, String nome, String email) {
        this.id = id;
        this.nome = nome;
        this.email = email;
        this.ativo = true;
    }
    
    @Override
    public Long getId() { return id; }
    
    @Override
    public String getNome() { return nome; }
    
    @Override
    public String getEmail() { return email; }
    
    @Override
    public boolean isAtivo() { return ativo; }
    
    public void ativar() { this.ativo = true; }
    public void inativar() { this.ativo = false; }
}

// Record para DTO (Java 16+)
public record ClienteDTO(
    Long id,
    String nome,
    String email,
    boolean ativo
) implements ICliente {
    // Já implementa todos os métodos automaticamente
}

// Uso consistente - ambos implementam ICliente
public class ClienteService {
    
    public void processarCliente(ICliente cliente) {
        System.out.println("Processando: " + cliente.getNomeFormatado());
        System.out.println("Email: " + cliente.getEmail());
    }
    
    // Pode receber tanto Cliente quanto ClienteDTO
    public void enviarEmail(ICliente cliente) {
        if (cliente.isAtivo()) {
            emailService.enviar(cliente.getEmail());
        }
    }
}

🔧 Técnicas de Refatoração

Extract Interface (Extrair Interface)

Crie uma interface comum que ambas as classes podem implementar. Unifique o uso.

Rename Method (Renomear Método)

Renomeie métodos para ter nomes consistentes entre classes similares.

Merge Classes (Mesclar Classes)

Se as classes são muito similares, considere mesclá-las em uma única classe.

Replace Type Code with State/Strategy

Use um enum ou tipo para diferenciar comportamentos ao invés de classes separadas.
