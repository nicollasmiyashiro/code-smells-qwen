Generalidade Especulativa

Speculative Generality

📖 Definição

Generalidade Especulativa ocorre quando código é criado para suportar funcionalidades futuras que nunca são implementadas. É o resultado de tentar prever demais o futuro e criar abstrações "úteis" que nunca são usadas.

Este smell viola o princípio YAGNI (You Ain't Gonna Need It): não implemente funcionalidades até que você realmente precise delas.

⚠️ Problemas Causados

  • Complexidade desnecessária: Código que ninguém usa.
  • Manutenção desperdiçada: Tempo gasto mantendo código inútil.
  • Confusão: Desenvolvedores não sabem se devem usar as abstrações.
  • Performance: Overhead de abstrações não utilizadas.
  • Oportunidade perdida: Tempo que poderia ser usado em features reais.

🔍 Como Identificar

  • Parâmetros que nunca são usados (sempre null ou valor padrão).
  • Métodos marcados como @Deprecated logo após criação.
  • Interfaces com implementações vazias ou dummy.
  • Comentários como "para uso futuro" ou "preparado para expansão".
  • Classes abstratas com apenas uma subclasse concreta.
  • Configurações que nunca são alteradas do padrão.

❌ Código RUIM

❌ Java - Generalidade Especulativa 
// Interface "preparada para o futuro"
public interface ProcessadorPagamento {
    boolean processar(Pagamento pagamento);
    boolean processarComReembolso(Pagamento pagamento, double valor);
    boolean processarParcial(Pagamento pagamento, int parcelas);
    boolean processarInternacional(Pagamento pagamento, Currency moeda);
    boolean processarRecorrente(Pagamento pagamento, int frequencia);
    void cancelar(String transacaoId);
    void estornar(String transacaoId, MotivoEstorno motivo);
}

// Implementação atual - só usa 2 métodos
public class ProcessadorPagamentoImpl implements ProcessadorPagamento {
    
    @Override
    public boolean processar(Pagamento pagamento) {
        // Implementação real
        return true;
    }
    
    @Override
    public boolean processarComReembolso(Pagamento pagamento, double valor) {
        // Nunca usado - sempre retorna false
        return false;
    }
    
    @Override
    public boolean processarParcial(Pagamento pagamento, int parcelas) {
        // Nunca usado - delega para processar normal
        return processar(pagamento);
    }
    
    @Override
    public boolean processarInternacional(Pagamento pagamento, Currency moeda) {
        // Nunca usado - não suportamos internacional
        throw new UnsupportedOperationException("Não suportado");
    }
    
    @Override
    public boolean processarRecorrente(Pagamento pagamento, int frequencia) {
        // Nunca usado
        throw new UnsupportedOperationException("Não suportado");
    }
    
    @Override
    public void cancelar(String transacaoId) {
        // Implementação real
    }
    
    @Override
    public void estornar(String transacaoId, MotivoEstorno motivo) {
        // Motivo nunca é usado
        estornar(transacaoId);
    }
    
    private void estornar(String transacaoId) {
        // Implementação real
    }
}

// Factory "preparada para múltiplos processadores"
public class ProcessadorFactory {
    
    private static Map processadores = new HashMap<>();
    
    static {
        // Só temos um processador, mas a estrutura está pronta para mais!
        processadores.put("CREDITO", new ProcessadorPagamentoImpl());
        processadores.put("DEBITO", new ProcessadorPagamentoImpl());
        // processadores.put("PIX", new ProcessadorPix()); // Futuro!
        // processadores.put("BOLETO", new ProcessadorBoleto()); // Futuro!
    }
    
    public static ProcessadorPagamento getProcessador(String tipo) {
        return processadores.getOrDefault(tipo, new ProcessadorPagamentoImpl());
    }
}

✅ Código BOM

✅ Java - YAGNI (You Ain't Gonna Need It) 
// Interface simples para o que realmente precisamos
public interface ProcessadorPagamento {
    boolean processar(Pagamento pagamento);
    void cancelar(String transacaoId);
    void estornar(String transacaoId);
}

// Implementação focada no necessário
public class ProcessadorPagamentoImpl implements ProcessadorPagamento {
    
    @Override
    public boolean processar(Pagamento pagamento) {
        // Implementação real do que usamos hoje
        validarPagamento(pagamento);
        conectarGateway();
        return confirmarTransacao();
    }
    
    @Override
    public void cancelar(String transacaoId) {
        // Implementação real
    }
    
    @Override
    public void estornar(String transacaoId) {
        // Implementação real
    }
}

// Uso direto, sem factory desnecessária
public class PedidoService {
    
    private final ProcessadorPagamento processador;
    
    // Injeção direta - se precisar de outro no futuro, mudamos aqui
    public PedidoService(ProcessadorPagamento processador) {
        this.processador = processador;
    }
    
    public void finalizarPedido(Pedido pedido) {
        processador.processar(pedido.getPagamento());
    }
    
    // Quando realmente precisarmos de PIX, adicionamos:
    // 1. Criamos ProcessadorPix implementando ProcessadorPagamento
    // 2. Injetamos ProcessadorPix onde necessário
    // 3. Sem factory, sem complexidade antecipada
}

🔧 Técnicas de Refatoração

Collapse Hierarchy (Colapsar Hierarquia)

Remova classes abstratas e interfaces que têm apenas uma implementação.

Remove Unused Parameters (Remover Parâmetros Não Usados)

Elimine parâmetros que sempre recebem o mesmo valor ou são ignorados.

Inline Method (Método Inline)

Substitua métodos que apenas delegam por seu conteúdo real.

Remove Dead Code (Remover Código Morto)

Delete métodos e classes que não são usados em nenhum lugar.

💡 Princípio YAGNI

YAGNI (You Ain't Gonna Need It) é um princípio do Extreme Programming que diz: "Sempre implemente as coisas quando você realmente precisar delas, nunca quando você apenas prevê que precisará."

Isso não significa não planejar. Significa não implementar baseado em previsões. Quando a necessidade surgir, você terá tempo de implementar adequadamente.
